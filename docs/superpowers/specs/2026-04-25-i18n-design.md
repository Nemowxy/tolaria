# i18n Design: English/Chinese Language Switching

## Overview

Add full internationalization to Tolaria, supporting English and Chinese with runtime switching. Covers all product-facing text: UI chrome, editor system messages, built-in templates, error messages, and command labels.

## Technical Stack

- **i18next** — core i18n framework
- **react-i18next** — React bindings (useTranslation hook, Trans component)
- **i18next-browser-languagedetector** — auto-detect system language on first launch

## Architecture

### Initialization Flow

```
App startup
  → Check Settings.language_preference (Tauri persisted)
  → If set: use it
  → If not set: i18next-browser-languagedetector reads navigator.language
    → zh-* → 'zh'
    → everything else → 'en'
  → i18next.init({ lng, resources })
  → React app renders with correct language
```

### i18n Configuration (`src/i18n.ts`)

- Initializes i18next with react-i18next
- Loads translation resources from `/locales/{en,zh}/translation.json`
- Fallback language: `en`
- Interpolation: `escapeValue: false` (React handles XSS)
- Language detector: checks Tauri Settings first, then navigator.language

### Translation Files

```
locales/
  en/
    translation.json
  zh/
    translation.json
```

Keys use semantic namespacing:

```json
{
  "settings": {
    "language": {
      "label": "Language",
      "english": "English",
      "chinese": "中文"
    },
    "theme": { "label": "Theme" }
  },
  "sidebar": {
    "projects": "Projects",
    "people": "People",
    "search": { "placeholder": "Search notes..." }
  },
  "editor": {
    "placeholder": "Start writing...",
    "errors": { "saveFailed": "Failed to save note" }
  },
  "commands": {
    "newNote": "New Note",
    "openVault": "Open Vault"
  },
  "errors": {
    "generic": "Something went wrong",
    "vaultNotFound": "Vault not found"
  },
  "search": {
    "results": "{{count}} result",
    "results_other": "{{count}} results",
    "noResults": "No results found"
  }
}
```

### Settings Extension

Add `language_preference` to the Settings type in `src/types.ts`:

```typescript
// In Settings interface
language_preference?: 'en' | 'zh';
```

Tauri backend `get_settings`/`save_settings` handles persistence automatically (serde deserializes the new field as optional).

### Language Switching

In SettingsPanel, add a language selector (shadcn/ui Select):

```
Language: [English ▾]  →  user picks 中文
  → i18next.changeLanguage('zh')
  → save to Settings via save_settings({ ...settings, language_preference: 'zh' })
  → all components using useTranslation() re-render
  → no restart required
```

## Scope

### In Scope (all translated)

- Settings panel labels and options
- Sidebar section names, buttons, tooltips
- Command palette labels and descriptions
- Search panel: placeholder, result counts, empty states
- Dialog titles, body text, button labels
- Toast/notification messages
- Editor placeholder text and system messages
- Error messages surfaced to the user (including Tauri command errors mapped on the frontend)
- Built-in templates and default content
- Status bar text
- Right-click context menus

### Out of Scope

- User-authored note content, file names, tags
- Git commit messages
- Third-party library internal error strings not exposed to users
- Tauri Rust-side log messages (not user-facing)

## Key Design Decisions

### Semantic keys, not English text as keys

`t('settings.language.label')` not `t('Language')`. Prevents ambiguity when the same English word maps to different Chinese translations in different contexts.

### Flat JSON with dot-notation grouping

Single `translation.json` per language (not split by namespace). The app is small enough that one file per language is manageable and simpler to maintain.

### i18next pluralization for counts

Use built-in plural rules: `results` / `results_other` keys. Chinese doesn't inflect for plurals but the structure stays consistent.

### Dynamic string concatenation → interpolation

Any existing pattern like `"Found " + count + " results"` must be converted to `t('search.results', { count })`.

### Tauri backend errors

Rust-side errors are returned as English strings. The frontend maps known error patterns to translation keys before displaying. Unknown errors fall back to a generic translated message + the original English in a detail expander.

## Component Migration Strategy

For every `.tsx`/`.ts` file containing user-facing strings:

1. Import `useTranslation` (or `t` from i18next for non-component code)
2. Replace each hardcoded string with `t('appropriate.key')`
3. Add the key + English value to `locales/en/translation.json`
4. Add the key + Chinese value to `locales/zh/translation.json`
5. Verify the component renders correctly in both languages

## Testing

### Automated

- Unit tests: i18n init logic, language detection fallback, language switching
- Integration tests: SettingsPanel language toggle changes visible text
- Existing tests: wrap test renders with `I18nextProvider` to prevent breakage

### Manual QA

- Full walkthrough in English, then switch to Chinese, verify every screen
- Check for layout overflow (Chinese can be wider or narrower than English depending on the phrase)
- Verify placeholder text in inputs, empty states, error dialogs
- Confirm language persists across app restart

### Residual String Check

After migration, run grep across `src/` for hardcoded English strings in JSX to catch any misses:

```bash
grep -rn '>[A-Z][a-z]' src/ --include='*.tsx' | grep -v 'import\|from\|//'
```

## Risks and Mitigations

| Risk | Mitigation |
|------|------------|
| Layout overflow with longer Chinese text | Manual QA pass; use flexible layouts (flex-wrap, min-width) |
| Missing translations at runtime | i18next fallback to English; CI check that both JSON files have identical key sets |
| Dynamic string concatenation missed | Grep for template literals and `+` concatenation in JSX |
| Tauri error messages not mapped | Catch-all: show generic translated error + English detail |
| Existing tests break without i18n provider | Add shared test utility that wraps components with I18nextProvider |
